# diagnose — 诊断循环

> 转自 [mattpocock/skills](https://github.com/mattpocock/skills)，版权归原作者所有。

**元数据**：面向复杂 bug 和性能回归的严谨诊断循环。

**触发时机**：用户说 "diagnose this" / "debug this"、报告 bug、说某物损坏/抛出异常/失败、或描述性能回归。

**描述**：复现 → 最小化 → 假设 → 仪器化 → 修复 → 回归测试。

---

## 诊断原则

这是一套针对复杂 bug 的纪律。**除非有明确理由，否则不要跳过阶段。**

探索代码库时，使用项目的领域词汇来建立相关模块的清晰心智模型，并检查你接触区域的 ADR。

## 阶段 1 — 构建反馈循环

**这才是核心技能。** 其他一切都是机械操作。如果你对 bug 有一个快速、确定性的、agent 可运行的通过/失败信号，你就能找到原因——二分法、假设测试和仪器化都只是在消费这个信号。如果没有这个信号，再怎么看代码也无济于事。

在这个阶段投入超比例的时间。**要激进、要有创意、绝不放弃。**

### 构建反馈循环的方法——大致按此顺序尝试

1. **失败的测试**——在能触及 bug 的任何接缝处：单元测试、集成测试、端到端测试。
2. **Curl / HTTP 脚本**——对着运行中的开发服务器发送请求。
3. **CLI 调用**——用 fixture 输入执行命令行，将输出与已知正确的快照对比。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，断言 DOM/控制台/网络行为。
5. **重放捕获的追踪**——保存真实的网络请求/载荷/事件日志到磁盘；在隔离环境中重放到相关代码路径。
6. **临时脚手架**——启动系统的最小集（一个服务，模拟依赖），用单次函数调用走通 bug 代码路径。
7. **属性/模糊测试循环**——如果 bug 是"有时输出错误"，跑 1000 个随机输入寻找失败模式。
8. **二分法脚手架**——如果 bug 出现在两个已知状态之间（提交、数据集、版本），自动化"在状态 X 启动、检查、重复"，这样可以用 `git bisect run`。
9. **差异循环**——对同一输入分别运行旧版本和新版本（或两个配置），对比输出。
10. **HITL bash 脚本**——最后手段。如果必须有人工操作，用 `scripts/hitl-loop.template.sh` 驱动他们，使循环仍然结构化。捕获的输出反馈给你。

构建出正确的反馈循环，bug 就已经解决了 90%。

### 迭代循环本身

把循环当成一个产品。一旦你有了**一个**循环，就问自己：

- 能更快吗？（缓存初始化、跳过无关初始化、缩小测试范围。）
- 信号能更精准吗？（断言具体症状，而不是"没有崩溃"。）
- 能更确定吗？（固定时间、播种 RNG、隔离文件系统、冻结网络。）

一个 30 秒的易碎循环比没有循环好不了多少。一个 2 秒的确定性循环就是调试超能力。

### 非确定性 bug

目标不是干净地复现，而是**提高复现率**。将触发器循环 100 次、并行化、增加压力、缩小时间窗口、注入睡眠。50% 随机性的 bug 是可调试的；1% 的则不行——持续提高复现率直到可调试。

### 确实无法构建循环时

停下来并明确说明。列出你尝试了什么。向用户请求：(a) 能复现的环境访问权限，(b) 捕获的工件（HAR 文件、日志转储、核心转储、带时间戳的屏幕录像），或 (c) 允许添加临时生产级仪器。**不要在没有循环的情况下进入假设阶段。**

## 阶段 2 — 复现

运行循环。观察 bug 出现。

确认：

- [ ] 循环产生的是**用户**描述的失败模式——不是附近碰巧发生的另一个失败。修错 bug = 修错问题。
- [ ] 失败在多次运行中可复现（或对于非确定性 bug，以足够高的比率复现以便调试）。
- [ ] 你已捕获了精确的症状（错误信息、错误输出、慢速计时），以便后续阶段验证修复是否真正解决了问题。

**在复现 bug 之前不要继续。**

## 阶段 3 — 假设

在测试任何假设之前，生成 **3-5 个排序后的假设**。单一假设会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它做出的预测。

> 格式："如果 <X> 是原因，那么 <修改 Y> 将使 bug 消失 / <修改 Z> 将使它更严重。"

如果你无法陈述预测，这个假设就是一种感觉—— discard 或 sharpen 它。

**在测试之前向用户展示排序列表。** 他们通常有领域知识可以立即重新排序（"我们刚对 #3 部署了一个变更"），或知道你已排除的假设。这是一个廉价的检查点，能大幅节省时间。如果用户不在线，按你自己的排序继续。

## 阶段 4 — 仪器化

每个探测必须对应阶段 3 中的一个具体预测。**每次只改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查**——如果环境支持。一个断点胜过十条日志。
2. **针对性日志**——放在能区分假设的边界处。
3. 永远不要"记录一切然后 grep"。

**为每条调试日志打上唯一前缀标签**，例如 `[DEBUG-a4f2]`。最后的清理就变成一次 grep。带标签的日志会被删除；不带标签的日志保留。

**性能分支。** 对于性能回归，日志通常不是正确的工具。应该：建立基线测量（计时脚手架、`performance.now()`、profiler、查询计划），然后二分。**先测量，再修复。**

## 阶段 5 — 修复 + 回归测试

**在修复之前**写回归测试——但前提是有一个**正确的接缝**。

正确的接缝是指测试能在调用点以 bug 实际发生的模式来锻炼**真正的 bug 模式**。如果唯一可用的接缝太浅（当 bug 需要多个调用者时的单调用者测试、无法复现触发 bug 的调用链的单元测试），在那里的回归测试只会带来虚假的信心。

**如果不存在正确的接缝，这本身就是发现。** 记录下来。代码库架构阻止了 bug 被锁定。将此标记为下一阶段的任务。

如果存在正确的接缝：

1. 将最小化的复现转化为该接缝处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 针对原始（未最小化的）场景重新运行阶段 1 的反馈循环。

## 阶段 6 — 清理 + 事后分析

在宣布完成之前必须完成：

- [ ] 原始复现不再复现（重新运行阶段 1 的循环）
- [ ] 回归测试通过（或不存在接缝已记录）
- [ ] 所有 `[DEBUG-...]` 仪器已删除（用前缀 grep）
- [ ] 临时原型已删除（或移至明确标记的调试位置）
- [ ] 最终正确的假设写入了提交/PR 信息——以便下一个调试者学习

**然后问：什么能防止这个 bug？** 如果答案涉及架构变更（没有好的测试接缝、调用者纠缠、隐藏的耦合），将具体信息移交给 `/improve-codebase-architecture` 技能。**在修复完成之后再提出建议**，而不是一开始——你现在拥有的信息比开始时更多。
