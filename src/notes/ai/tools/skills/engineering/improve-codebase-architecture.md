# improve-codebase-architecture — 改善代码库架构

> 转自 [mattpocock/skills](https://github.com/mattpocock/skills)，版权归原作者所有。

**触发时机**：用户想要改善架构、寻找重构机会、整合紧耦合模块、或让代码库更易于测试和 AI 导航。

**描述**：发现深化机会——将浅模块转变为深模块的重构，以提升可测试性和 AI 可导航性。

---

## 术语表

在每条建议中**精确使用这些术语**。一致的语言就是重点——不要漂移成"组件"、"服务"、"API"或"边界"。

- **Module（模块）** — 任何有接口和实现的东西（函数、类、包、切片）。
- **Interface（接口）** — 调用者使用模块必须知道的一切：类型、不变量、错误模式、顺序、配置。不只是类型签名。
- **Implementation（实现）** — 内部的代码。
- **Depth（深度）** — 接口处的杠杆率：小接口背后有大量行为。**深** = 高杠杆。**浅** = 接口几乎与实现一样复杂。
- **Seam（接缝）** — 接口所在之处；可以在不就地修改的情况下改变行为的位置。（用这个词，不用"边界"。）
- **Adapter（适配器）** — 在接缝处满足接口的具体事物。
- **Leverage（杠杆）** — 调用者从深度中得到的收益。
- **Locality（局部性）** — 维护者从深度中得到的收益：变更、bug、知识集中在一处。

关键原则：

- **删除测试**：想象删除这个模块。如果复杂度消失了，它只是个透传。如果复杂度在 N 个调用者中重新出现，它就在创造价值。
- **接口就是测试面。**
- **一个适配器 = 假设的接缝。两个适配器 = 真正的接缝。**

这个技能受项目的领域模型**启发**。领域语言为好接缝命名；ADR 记录技能不应重新辩论的决策。

## 流程

### 1. 探索

先阅读项目的领域词汇表和你接触区域的 ADR。

然后用 Agent 工具（`subagent_type=Explore`）有机地探索代码库。不要遵循僵化的启发式——有机地探索，注意你在哪里遇到了摩擦：

- 理解一个概念需要在许多小模块之间跳来跳去？
- 哪些模块是**浅**的——接口几乎与实现一样复杂？
- 哪些纯函数只是为了可测试性被提取出来，但真正的 bug 隐藏在它们的调用方式中（缺乏**局部性**）？
- 哪些紧耦合模块在接缝处互相泄漏？
- 代码库的哪些部分未被测试，或通过当前接口难以测试？

对你怀疑浅薄的事物应用**删除测试**：删除它会使复杂度集中还是只是转移？"是的，会集中"就是你要的信号。

### 2. 将候选项呈现为 HTML 报告

将报告写到操作系统的临时目录中，这样不会污染仓库。从 `$TMPDIR` 解析临时目录，回退到 `/tmp`（Windows 上为 `%TEMP%`），写入 `<tmpdir>/architecture-review-<timestamp>.html`，这样每次运行都得到新文件。为用户打开它——Windows 上用 `start <path>`，macOS 上用 `open <path>`，Linux 上用 `xdg-open <path>`——并告诉用户绝对路径。

报告使用 **Tailwind CDN** 进行布局和样式，使用 **Mermaid CDN** 绘制图表。Mermaid 处理图形关系的图（调用图、依赖、时序），手工构建的 div 和 SVG 处理更具编辑感观的视觉（质量图、截面图、折叠动画）。混合使用——不要全都用 Mermaid，那样看起来会很通用。

每个候选项的卡片模板：

- **Files（文件）** — 涉及哪些文件/模块
- **Problem（问题）** — 为什么当前架构在制造摩擦
- **Solution（方案）** — 用通俗语言描述会改变什么
- **Benefits（收益）** — 用局部性和杠杆率来解释，以及测试会如何改善
- **Before / After 图** — 并排展示，手绘风格，说明浅薄和深化
- **推荐强度** — `Strong`（强烈推荐）、`Worth exploring`（值得探索）、`Speculative`（推测性），显示为徽章

报告末尾加一个 **Top recommendation（首要推荐）** 部分：你最先处理哪个候选项及原因。

**使用 CONTEXT.md 的词汇描述领域，使用 LANGUAGE.md 的词汇描述架构。**

**ADR 冲突**：如果候选项与现有 ADR 矛盾，只有在摩擦足够大值得重新审视 ADR 时才展示。在卡片中明确标记（例如警告标注：*"与 ADR-0007 矛盾——但值得重新讨论，因为……"*）。不要列出 ADR 禁止的每个理论重构。

**不要立即提议接口。** 文件写完后，问用户："你想深入探索哪个？"

### 3. 拷问循环

用户选择候选项后，进入拷问对话。与他们一起走设计树——约束、依赖、深化模块的形状、接缝后面有什么、哪些测试能保留。

决策明确时产生副作用：

- **以不在 `CONTEXT.md` 中的概念命名深化后的模块？** 将术语添加到 `CONTEXT.md`。
- **对话中提炼了模糊术语？** 就地更新 `CONTEXT.md`。
- **用户以有分量的理由拒绝了候选项？** 提议创建 ADR， framed 为：*"要我把它记录为 ADR 吗？这样以后的架构审查不会再次建议它。"*
- **想探索深化模块的替代接口？** 参见 INTERFACE-DESIGN.md。
