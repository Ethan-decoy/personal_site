# prototype — 原型

> 转自 [mattpocock/skills](https://github.com/mattpocock/skills)，版权归原作者所有。

**触发时机**：用户想进行原型设计、对数据模型/状态机做合理性检查、mock UI、探索设计选项，或说"原型化这个"、"让我玩玩看"、"试几种设计"。

**描述**：构建一次性原型来探索设计方向——可以是终端应用来验证状态/业务逻辑，也可以是多个截然不同的 UI 变体通过一个路由切换。

---

原型是**一次性代码，用来回答一个问题**。问题决定形态。

## 选择分支

识别正在回答哪个问题——从用户提示、周围代码、或在用户在时提问：

- **"这个逻辑/状态模型感觉对吗？"** → 走 LOGIC 分支。构建一个小型交互式终端应用，把状态机推到在纸上难以推理的边界情况中。
- **"这应该长什么样？"** → 走 UI 分支。在单个路由上生成几个截然不同的 UI 变体，通过 URL 搜索参数和悬浮底部栏切换。

两个分支产出非常不同的产物——搞错会浪费整个原型。如果问题确实模棱两可且用户联系不上，默认选择更符合周围代码的分支（后端模块 → 逻辑；页面或组件 → UI）并在原型顶部声明假设。

## 通用规则

1. **从第一天起就是临时性的，并明确标记。** 将原型代码放在它将实际被使用的位置附近（紧邻它原型的模块或页面），这样上下文显而易见——但命名要让随意阅读的人能看出它是原型而非生产代码。对于临时 UI 路由，遵守项目已有的路由约定；不要发明新的顶级结构。

2. **一条命令运行。** 无论项目现有的任务运行器支持什么——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能不假思索地启动它。

3. **默认无持久化。** 状态驻留在内存中。持久化是原型正在**检查**的东西，不是它应该依赖的东西。如果问题显式涉及数据库，使用一个临时 DB 或带有清晰"原型——可删除"名称的本地文件。

4. **跳过打磨。** 不要测试、不要错误处理（除了让原型可运行的部分）、不要抽象。重点是快速学到东西然后删掉它。

5. **暴露状态。** 每个动作之后（逻辑）或每次变体切换（UI），打印或渲染完整的状态，让用户看到什么发生了变化。

6. **完成后删除或吸收。** 当原型回答了它的问题后，要么删掉它，要么将验证过的决策合并到真实代码中——不要让它烂在仓库里。

## 完成后

唯一值得保留的是原型的**答案**。将它捕获到持久的地方（提交信息、ADR、issue、或原型旁边的 `NOTES.md`），以及它回答的问题。如果用户在线，这是一个快速对话；如果不在线，留下占位符以便他们（或你在下一轮）在删除原型前填写结论。
