# tdd — 测试驱动开发

> 转自 [mattpocock/skills](https://github.com/mattpocock/skills)，版权归原作者所有。

**触发时机**：用户想用 TDD 构建功能或修复 bug、提到"红-绿-重构"、想要集成测试、或请求测试优先开发。

**描述**：红-绿-重构循环的测试驱动开发。一次一个垂直切片来构建功能或修复 bug。

---

## 哲学

**核心原则**：测试应该通过公共接口验证行为，而不是实现细节。代码可以完全改变；测试不应该。

**好测试**是集成风格的：通过公共 API 锻炼真实代码路径。它们描述系统**做什么**，而不是**怎么做**。好测试读起来像规格说明——"用户可以凭有效购物车结账"准确告诉你存在什么能力。这些测试能在重构中存活，因为它们不关心内部结构。

**坏测试**与实现耦合。它们模拟内部协作者、测试私有方法、或通过外部手段验证（比如直接查询数据库而不是使用接口）。警告信号：你重构时测试断了，但行为没有变化。如果你重命名一个内部函数而测试失败，那些测试测的是实现而不是行为。

## 反模式：水平切片

**不要先写所有测试，再写所有实现。** 这是"水平切片"——把 RED 当成"写所有测试"、GREEN 当成"写所有代码"。

这会产生**糟糕的测试**：

- 批量写出的测试测的是*想象中的*行为，不是*实际的*行为
- 你最终测的是事物的*形状*（数据结构、函数签名）而不是面向用户的行为
- 测试对真实变化不敏感——行为坏了测试通过，行为正常测试失败
- 你跑在了 headlights 前面，在理解实现之前就承诺了测试结构

**正确做法**：通过示踪弹进行垂直切片。一个测试 → 一个实现 → 重复。每个测试回应你从上一个周期中学到的东西。因为你刚写了代码，你确切知道什么行为重要以及如何验证它。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流

### 1. 规划

探索代码库时，使用项目的领域词汇，使测试名称和接口词汇与项目语言匹配，并尊重你接触区域的 ADR。

在写任何代码之前：

- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认要测试哪些行为（按优先级）
- [ ] 寻找深模块的机会（小接口、深实现）
- [ ] 为可测试性设计接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户对计划的批准

问："公共接口应该长什么样？哪些行为最重要需要测试？"

**你无法测试一切。** 与用户确认哪些行为最重要。将测试精力集中在关键路径和复杂逻辑上，而不是每个可能的边界情况。

### 2. 示踪弹

写**一个**测试来确认系统中的**一件事**：

```
RED:   为第一个行为写测试 → 测试失败
GREEN: 写最少量代码使其通过 → 测试通过
```

这是你的示踪弹——证明端到端的路径是通的。

### 3. 增量循环

对每个剩余行为：

```
RED:   写下一个测试 → 失败
GREEN: 最少量代码使其通过 → 通过
```

规则：

- 一次一个测试
- 只写足够通过当前测试的代码
- 不要预期未来的测试
- 保持测试聚焦于可观察行为

### 4. 重构

所有测试通过后，寻找重构候选项：

- [ ] 提取重复代码
- [ ] 深化模块（将复杂度移到简单接口后面）
- [ ] 在自然的地方应用 SOLID 原则
- [ ] 考虑新代码揭示了现有代码的什么
- [ ] 每次重构步骤后运行测试

**RED 时永远不要重构。** 先到 GREEN。

## 每个周期的检查清单

```
[ ] 测试描述行为，不是实现
[ ] 测试只使用公共接口
[ ] 测试能在内部重构后存活
[ ] 代码对当前测试是最少的
[ ] 没有投机性特性
```
