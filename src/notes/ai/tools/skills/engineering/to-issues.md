# to-issues — 拆分为 Issue

> 转自 [mattpocock/skills](https://github.com/mattpocock/skills)，版权归原作者所有。

**触发时机**：用户想将计划转换为 issue、创建实现工单、或将工作拆分为 issue。

**描述**：使用示踪弹（tracer bullet）垂直切片将计划、规格或 PRD 拆分为可独立获取的 issue。

---

## 流程

### 1. 收集上下文

从对话上下文中已有的内容出发。如果用户传递了 issue 引用（issue 号、URL、或路径）作为参数，从问题追踪器获取它并阅读完整正文和评论。

### 2. 探索代码库（可选）

如果还没有探索过代码库，进行探索以了解当前状态。Issue 标题和描述应使用项目领域词汇，并尊重相关区域的 ADR。

### 3. 起草垂直切片

将计划拆分为**示踪弹** issue。每个 issue 是一个薄的垂直切片，端到端穿过**所有**集成层，而不是某一层的水平切片。

切片可以是 'HITL'（需要人工交互）或 'AFK'（无需人工交互即可实现和合并）。HITL 切片需要人工交互，例如架构决策或设计审查。AFK 切片可以在没有人工交互的情况下实现和合并。尽可能优先 AFK。

**垂直切片规则**：
- 每个切片在每个层（schema、API、UI、测试）提供一个窄但**完整**的路径
- 完成的切片可以独立演示或验证
- 宁可多薄不要多厚

### 4. 向用户提问

将提议的分解展示为编号列表。对每个切片展示：

- **标题**：简短描述性名称
- **类型**：HITL / AFK
- **被阻塞于**：哪些其他切片（如果有）必须先完成
- **覆盖的用户故事**：这解决了哪些用户故事（如果源材料有的话）

询问用户：

- 粒度感觉对吗？（太粗 / 太细）
- 依赖关系正确吗？
- 是否有切片应该合并或进一步拆分？
- 正确的切片被标记为 HITL 和 AFK 了吗？

迭代直到用户批准分解。

### 5. 将 issue 发布到问题追踪器

对每个批准的切片，发布新 issue 到问题追踪器。使用下方的 issue 正文模板。这些 issue 被认为已准备好给 AFK agent，所以用正确的分派标签发布它们，除非另有指示。

按依赖顺序发布 issue（阻塞者先），这样你可以在"被阻塞于"字段中引用真实的 issue 标识符。

#### Issue 模板

```markdown
## Parent

对问题追踪器上父 issue 的引用（如果源是已有 issue，否则省略）。

## 要构建什么

这个垂直切片的简洁描述。描述端到端的行为，而不是逐层的实现。

避免具体的文件路径或代码片段——它们很快过时。例外：如果原型产出的片段能以比文字更精确的方式编码决策（状态机、reducer、schema、类型形状），在此内联它并简短注明来自原型。裁剪到富含决策的部分——不是可运行的 demo，只是重要的部分。

## 验收标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 被阻塞于

- 对阻塞工单的引用（如果有）

或"无 - 可立即开始"如果没有阻塞。
```

**不要**关闭或修改任何父 issue。
