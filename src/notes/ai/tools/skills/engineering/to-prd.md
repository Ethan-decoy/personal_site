# to-prd — 生成 PRD

> 转自 [mattpocock/skills](https://github.com/mattpocock/skills)，版权归原作者所有。

**触发时机**：用户想从当前对话上下文创建 PRD。

**描述**：将当前对话上下文转化为 PRD 并发布到项目问题追踪器。不需要访谈——仅综合你已经讨论过的内容。

---

**不要采访用户**——只需综合你已经知道的东西。

问题追踪器和分派标签词汇应该已提供给你——如果没有，运行 `/setup-matt-pocock-skills`。

## 流程

1. 如果还没做过，探索仓库以了解代码库当前状态。在整个 PRD 中使用项目领域词汇，并尊重你接触区域的任何 ADR。

2. 勾勒你需要构建或修改的主要模块来完成实现。积极寻找可以提取为**深模块**的机会，这些模块可以独立测试。

深模块（相对于浅模块）是指封装了大量功能到一个简单、可测试、很少变化的接口中。

与用户确认这些模块是否符合他们的期望。与用户确认他们想为哪些模块编写测试。

3. 使用下方模板编写 PRD，然后发布到项目问题追踪器。应用 `ready-for-agent` 分派标签——不需要额外分派。

## PRD 模板

```markdown
## Problem Statement

用户面临的问题，从用户的角度。

## Solution

问题的解决方案，从用户的角度。

## User Stories

一个**很长的**编号用户故事列表。每个用户故事格式为：

1. 作为 <角色>，我想要 <功能>，以便 <收益>

示例：
1. 作为手机银行客户，我想看到账户余额，以便我能做出更明智的消费决策

这个用户故事列表应该非常全面，涵盖功能的所有方面。

## Implementation Decisions

做出的实现决策列表。可以包括：

- 将要构建/修改的模块
- 将要修改的那些模块的接口
- 开发者的技术澄清
- 架构决策
- Schema 变更
- API 合约
- 特定交互

**不要**包含具体的文件路径或代码片段。它们很快会过时。

例外：如果原型产出的片段能以比文字更精确的方式编码决策（状态机、reducer、schema、类型形状），在相关决策中内联它并简短注明来自原型。

## Testing Decisions

做出的测试决策列表。包括：

- 什么构成好测试的描述（只测试外部行为，不测试实现细节）
- 哪些模块将被测试
- 测试的先例（即代码库中类似的测试类型）

## Out of Scope

对此 PRD 来说超出范围的内容描述。

## Further Notes

关于功能的任何进一步说明。
```
