# triage — 问题分派

> 转自 [mattpocock/skills](https://github.com/mattpocock/skills)，版权归原作者所有。

**触发时机**：用户想要创建 issue、分派 issue、审查 incoming bug/需求、为 AFK agent 准备 issue、或管理工作流。

**描述**：通过状态机驱动的分派角色流程来处理问题。

---

## 重要规则

分派期间发布到问题追踪器的每条评论或 issue **必须**以以下声明开头：

> *This was generated by AI during triage.*

## 角色

两个**类别**角色：

- `bug` — 某物损坏了
- `enhancement` — 新功能或改进

五个**状态**角色：

- `needs-triage` — 维护者需要评估
- `needs-info` — 等待报告人提供更多信息
- `ready-for-agent` — 已完整指定，可交给 AFK agent
- `ready-for-human` — 需要人工实现
- `wontfix` — 不会采取行动

每个已分派的问题应恰好携带一个类别角色和一个状态角色。如果状态角色冲突，在进行任何其他操作之前标记并询问维护者。

这些是规范角色名——问题追踪器中使用的实际标签字符串可能不同。映射应该已提供给你——如果没有，运行 `/setup-matt-pocock-skills`。

**状态流转**：未标记的问题先走 `needs-triage`；从那里转到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。报告人回复后，`needs-info` 回到 `needs-triage`。维护者可以随时覆盖——标记看起来异常的流转并在继续之前询问。

## 调用方式

维护者调用 `/triage` 并用自然语言描述他们想要的。解释并执行。示例：

- "让我看看任何需要我注意的东西"
- "看看 #42"
- "把 #42 移到 ready-for-agent"
- "有什么可以被 agent 接手的？"

## 显示需要关注的内容

查询问题追踪器，按时间从早到晚展示三个桶：

1. **未标记** — 从未分派过。
2. **`needs-triage`** — 评估中。
3. **`needs-info` 且报告人在上次分派笔记后有活动** — 需要重新评估。

展示每个桶的数量和每个 issue 的一行摘要。让维护者选择。

## 分派具体问题

1. **收集上下文。** 阅读完整 issue（正文、评论、标签、报告人、日期）。解析之前的分派笔记，避免重新问已解决的问题。使用项目领域词汇探索代码库，尊重相关区域的 ADR。阅读 `.out-of-scope/*.md` 并展示任何与当前 issue 类似的先前拒绝记录。

2. **推荐。** 告诉维护者你的类别和状态推荐及推理，加上与 issue 相关的代码库简要概述。等待指示。

3. **复现（仅 bug）。** 在任何拷问之前，尝试复现：阅读报告人的步骤、追踪相关代码、运行测试或命令。报告发生了什么——成功复现并给出代码路径、复现失败、或细节不足（强烈的 `needs-info` 信号）。确认的复现能生成更好的 agent brief。

4. **拷问（如需要）。** 如果 issue 需要充实，运行 `/grill-with-docs` 会话。

5. **应用结果：**
   - `ready-for-agent` — 发布 agent brief 评论。
   - `ready-for-human` — 结构与 agent brief 相同，但注明为什么不能委托（判断决策、外部访问、设计决策、手动测试）。
   - `needs-info` — 发布分派笔记（见下方模板）。
   - `wontfix`（bug）— 礼貌解释，然后关闭。
   - `wontfix`（enhancement）— 写入 `.out-of-scope/`，在评论中链接到它，然后关闭。
   - `needs-triage` — 应用角色。如有部分进展可加可选评论。

## 快速状态覆盖

如果维护者说"把 #42 移到 ready-for-agent"，信任他们并直接应用角色。确认你即将做什么（角色变更、评论、关闭），然后执行。跳过拷问。如果在没有拷问会话的情况下移到 `ready-for-agent`，询问他们是否想写 agent brief。

## Needs-info 模板

```markdown
## Triage Notes

**目前已确认的：**

- 第 1 点
- 第 2 点

**我们仍需要你 (@reporter) 提供：**

- 问题 1
- 问题 2
```

在"目前已确认的"下捕获拷问期间解决的所有内容，这样工作不会丢失。问题必须具体且可操作，不能是"请提供更多信息"。

## 恢复之前的会话

如果 issue 上已有分派笔记，阅读它们，检查报告人是否回答了任何未解决的问题，在继续之前展示更新后的全貌。不要重新问已解决的问题。
