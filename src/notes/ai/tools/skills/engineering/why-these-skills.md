# 为什么需要这些技能

> 转自 [mattpocock/skills](https://github.com/mattpocock/skills)，版权归原作者所有。

我构建这些技能是为了解决我在使用 Claude Code、Codex 等编程 agent 时常见的失败模式。

## #1：Agent 没有做出我想要的东西

> "没有人能确切地知道自己想要什么"
>
> David Thomas & Andrew Hunt，《[程序员修炼之道](https://www.amazon.co.uk/Pragmatic-Programmer-Anniversary-Journey-Mastery/dp/B0833F1T3V)》

**问题**：软件开发中最常见的失败模式是**对齐偏差**。你以为开发者理解你的需求，然后你看到他们做出的东西——才发现他们完全没有理解你。

AI 时代也是一样的。你和 agent 之间存在沟通鸿沟。解决办法是进行一次 **"拷问式"对话**（grilling session）——让 agent 就你要构建的东西向你提出详细的问题。

**解决方法**是使用：

- `/grill-me` — 适用于非代码场景
- `/grill-with-docs` — 与 `/grill-me` 类似，但增加了更多功能

这是我最受欢迎的两个技能。它们帮助你在动手之前与 agent 达成对齐，并深入思考你要做的变更。**每次**要做变更时都应该使用它们。

## #2：Agent 的表达过于冗长

> 在统一语言中，开发者之间的对话和代码的表达都源自同一个领域模型。
>
> Eric Evans，《[领域驱动设计](https://www.amazon.co.uk/Domain-Driven-Design-Tackling-Complexity-Software/dp/0321125215)》

**问题**：在项目初期，开发者和他们为之构建软件的人（领域专家）通常说的是不同的语言。

我的 agent 也存在同样的张力。Agent 通常被丢进一个项目，然后被要求自行理解术语。所以它们会用 20 个词来表达 1 个词就能说清的事。

**解决方法**是建立一套**共享语言**。它是一份文档，帮助 agent 解码项目中使用的行话。

### 示例

这是一个 `CONTEXT.md` 的示例，来自我的 `course-video-manager` 仓库。哪个更容易理解？

- **修改前**："当课程中某个章节内的课时被设为'真实'（即在文件系统中获得一个位置）时出现了问题"
- **修改后**："物化级联出现了问题"

这种简洁性在每次会话中都在产生回报。

这个机制已经内置到 `/grill-with-docs` 中。它是一次拷问式对话，但同时帮助你与 AI 建立共享语言，并将难以解释的决策记录在 ADR（架构决策记录）中。

很难解释这个技能有多强大。它可能是这个仓库里最酷的技术。试试看。

> [!TIP]
> 共享语言还有诸多好处，不止于减少冗长表达：
>
> - **变量、函数和文件的命名保持一致**，基于共享语言
> - 因此，**代码库对 agent 来说更容易导航**
> - Agent 的 **thinking token 消耗也更少**，因为它可以访问更简洁的语言

## #3：代码跑不起来

> "始终采取小而谨慎的步骤。反馈的速度就是你的速度上限。永远不要承担太大的任务。"
>
> David Thomas & Andrew Hunt，《[程序员修炼之道](https://www.amazon.co.uk/Pragmatic-Programmer-Anniversary-Journey-Mastery/dp/B0833F1T3V)》

**问题**：假设你和 agent 就构建目标达成了一致。但 agent **仍然**产出了糟糕的代码怎么办？

该审视你的**反馈循环**了。如果没有反馈来验证它产出的代码实际运行情况，agent 就是在盲飞。

**解决方法**：你需要一组标准的反馈循环——静态类型检查、浏览器访问和自动化测试。

对于自动化测试，红-绿-重构循环至关重要。即 agent 先写一个失败的测试，然后修复它直到通过。这为 agent 提供了一致的反馈机制，从而产生更好的代码。

我构建了一个 **`/tdd` 技能**，可以嵌入任何项目。它鼓励红-绿-重构实践，并为 agent 提供了大量关于什么是好测试和坏测试的指导。

对于调试，我还构建了一个 **`/diagnose`** 技能，将最佳调试实践封装在一个简单的循环中。

## #4：我们构建了一坨泥球（Ball of Mud）

> "每天都要投资于系统的设计。"
>
> Kent Beck，《[极限编程解析](https://www.amazon.co.uk/Extreme-Programming-Explained-Embrace-Change/dp/0321278658)》

> "最好的模块是'深'的。它们允许通过简单的接口访问大量功能。"
>
> John Ousterhout，《[软件设计的哲学](https://www.amazon.co.uk/Philosophy-Software-Design-2nd/dp/173210221X)》

**问题**：大多数用 agent 构建的应用都很复杂且难以变更。因为 agent 能极大地加速编码，所以它也加速了软件的熵增。代码库的复杂度以前所未有的速度增长。

**解决方法**是一种全新的 AI 驱动开发方式：**关心代码设计**。

这内置于这些技能的每一层：

- `/to-prd` 在创建 PRD 之前会询问你涉及了哪些模块
- `/zoom-out` 让 agent 在整个系统的上下文中解释代码

而关键的是，`/improve-codebase-architecture` 帮助你挽救一个已经变成泥球的代码库。我建议你每隔几天就在代码库上运行一次。

## 总结

软件工程的基本原理比以往任何时候都更重要。这些技能是我将这些基本原理浓缩为可重复实践的最大努力，帮助你打造出职业生涯中最好的应用。
